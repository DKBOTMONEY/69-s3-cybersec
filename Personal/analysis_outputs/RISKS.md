# Marquee Implementation Risks & Audit (การวิเคราะห์ความเสี่ยงของส่วนประกอบ Marquee)

การวิเคราะห์ความเสี่ยงและแนวทางความปลอดภัยในการใช้คอมโพเนนต์วิ่งตัวอักษร/ภาพ (Marquee Component) บนระบบ Next.js:

---

## ⚠️ 1. ความเสี่ยงด้านประสิทธิภาพยามใช้งาน (Performance & Layout Thrashing Risk)

*   **ปัญหาตรวจพบ**: โค้ดของ Marquee มีการเรียกใช้คำสั่งอ่านค่าคำนวณสไตล์ความกว้างและ Gap ทุกๆ เฟรมภาพ (60 ครั้งต่อวินาที) ผ่านทางคำสั่งนี้ใน `useAnimationFrame`:
    ```typescript
    const singleContentBlockSize = vertical ? singleContentBlockRef.current.offsetHeight : singleContentBlockRef.current.offsetWidth;
    const contentStyle = window.getComputedStyle(contentRef.current);
    ```
*   **ผลกระทบ**: การอ่านค่าขนาดกว้าง/สูง (`offsetWidth`/`offsetHeight`) และการคำนวณสไตล์จริง (`window.getComputedStyle`) ในขณะที่หน้าจอกำลังอัปเดตสไตล์ตำแหน่งผ่าน CSS `transform` จะส่งผลให้เบราว์เซอร์เกิดการทำ **Reflow (Layout Thrashing)** ซ้ำไปซ้ำมา ส่งผลให้อัตราเฟรมเรตตก การเลื่อนสะดุด หรือกินหน่วยประมวลผล CPU สูงมากในเครื่องที่สเปกไม่แรง
*   **แนวทางแก้ไขที่แนะนำ**:
    *   ควรวัดขนาดกว้าง/สูงและช่องว่าง Gap **เพียงครั้งแรกที่เมานต์คอมโพเนนต์ (onMount)** และอัปเดตใหม่เฉพาะเวลาที่ผู้ใช้ย่อขยายหน้าจอเบราว์เซอร์ (onResize) เท่านั้น แล้วบันทึกระยะดังกล่าวลงในตัวแปรอ้างอิง `useRef` หรือ `useState` แทนการวัดใหม่ทุกเฟรมภาพภาพอนิเมชัน

---

## ❄️ 2. ความเสี่ยงยามแสดงผลบนเซิร์ฟเวอร์ (Hydration / SSR Mismatch Risk)

*   **ปัญหาตรวจพบ**: ตัวระบบของ Next.js จะพยายามเรนเดอร์โครงสร้าง HTML ในฝั่งเซิร์ฟเวอร์ก่อนส่งให้ผู้ใช้ (Server-Side Rendering)
*   **ผลกระทบ**: เนื่องจาก Marquee จะต้องรอกลไกคำนวณค่ากว้าง/สูงของเอเลเมนต์หลังเปิดหน้าจอจริง การแสดงผลอาจมีการกระตุกเล็กน้อย หรือสัดส่วนกระโดดขยับ (Visual Layout Shift) ในจังหวะที่หน้าเว็บเริ่มโหลดข้อมูลในหน้าเบราว์เซอร์ครั้งแรก
*   **แนวทางแก้ไข**: ควรใส่กลไกเช็คความพร้อมการเมานต์ฝั่งไคลเอนต์ (`mounted` state) เพื่อให้ระบบรันทำงานอนิเมชันเฉพาะหลังจากที่เกิดการเชื่อมต่อไคลเอนต์ (Hydrated) สมบูรณ์แล้วเท่านั้น
